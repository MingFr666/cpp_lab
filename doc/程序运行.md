# C/C++ 程序编译过程、静态库与动态库总结

## 1. 程序从源码到可执行文件的整体过程

C/C++ 程序从源码到可执行文件，通常经历四个主要阶段：

```text
源代码 .cpp
  ↓ 预处理
预处理后源码 .i
  ↓ 编译
汇编代码 .s
  ↓ 汇编
目标文件 .o / .obj
  ↓ 链接
可执行文件 / 动态库 / 静态库
```

常见分步命令：

```bash
# 1. 预处理
g++ -E main.cpp -o main.i

# 2. 编译成汇编
g++ -S main.i -o main.s

# 3. 汇编成目标文件
g++ -c main.s -o main.o

# 4. 链接成可执行文件
g++ main.o -o main
```

平时直接写：

```bash
g++ main.cpp -o main
```

其实是编译器自动完成了以上几个阶段。

---

## 2. 预处理阶段

预处理阶段主要处理 `#` 开头的预处理指令。

### 2.1 头文件展开

```cpp
#include "add.h"
```

预处理器会把 `add.h` 的内容直接拷贝到当前 `.cpp` 文件中。

所以从编译器视角看，每个 `.cpp` 在编译前都会变成一个独立的、完整的文本文件。

这个完整的编译单元叫做：

```text
翻译单元 Translation Unit
```

### 2.2 宏替换

```cpp
#define MAXN 100
```

预处理时会被直接替换成：

```cpp
100
```

宏不是变量，也不是函数，本质是文本替换。

### 2.3 条件编译

```cpp
#ifdef DEBUG
std::cout << "debug mode" << std::endl;
#endif
```

根据宏是否定义，决定某段代码是否参与编译。

### 2.4 头文件保护

常见写法：

```cpp
#ifndef ADD_H
#define ADD_H

int add(int a, int b);

#endif
```

或者：

```cpp
#pragma once
```

作用是防止同一个头文件在同一个翻译单元中被重复包含。

注意：

```text
头文件保护只能防止同一个 .cpp 中重复 include；
不能防止多个 .cpp 中重复定义同一个普通函数。
```

例如：

```cpp
// add.h
int add(int a, int b) {
    return a + b;
}
```

如果多个 `.cpp` 都 include 这个头文件，每个 `.cpp` 都会生成一个 `add` 的定义，链接时可能报：

```text
multiple definition of `add(int, int)`
```

解决方法：

```cpp
// add.h
int add(int a, int b);
```

```cpp
// add.cpp
int add(int a, int b) {
    return a + b;
}
```

或者把函数声明为 `inline`：

```cpp
inline int add(int a, int b) {
    return a + b;
}
```

## 3. 编译阶段

编译阶段把预处理后的 C++ 代码转换成汇编代码。

主要工作包括：

### 3.1 语法检查

例如：

```cpp
int x = "abc";
```

类型不匹配，编译阶段会报错。

### 3.2 语义检查

例如：

```cpp
class A {
private:
    int x;
};

int main() {
    A a;
    a.x = 1;
}
```

访问 private 成员，编译阶段会报错。

### 3.3 生成汇编代码

例如：

```cpp
int add(int a, int b) {
    return a + b;
}
```

可能生成类似汇编：

```asm
add:
    mov eax, edi
    add eax, esi
    ret
```

### 3.4 编译优化

例如：

```cpp
int x = 1 + 2;
```

编译器可能直接优化成：

```cpp
int x = 3;
```

函数是否真正内联展开，也主要由编译器优化器决定。

`inline` 在现代 C++ 中更重要的意义是：

```text
允许函数在多个翻译单元中重复定义，只要定义完全相同。
```

是否真的展开函数调用，是优化器决定的。

---

## 4. 汇编阶段

汇编阶段把 `.s` 汇编代码转换为机器码，生成目标文件：

```text
.o / .obj
```

目标文件已经是二进制文件，但还不能直接运行。

原因是它可能还有一些未解决的符号。

例如：

```cpp
// main.cpp
extern int add(int, int);

int main() {
    return add(1, 2);
}
```

编译成 `main.o` 后，目标文件中会记录：

```text
我需要一个叫 add(int, int) 的函数
```

但它不知道 `add` 的实现在哪里。

这个问题要在链接阶段解决。

---

## 5. 链接阶段

链接阶段的核心任务：

```text
把多个目标文件和库文件合并，解决符号引用，生成最终可执行文件或动态库。
```

例如：

```bash
g++ -c main.cpp -o main.o
g++ -c add.cpp -o add.o
g++ main.o add.o -o main
```

其中：

```cpp
// main.cpp
int add(int, int);

int main() {
    return add(1, 2);
}
```

```cpp
// add.cpp
int add(int a, int b) {
    return a + b;
}
```

`main.o` 中引用了 `add`，`add.o` 中定义了 `add`。

链接器会把它们合并，并把 `main.o` 中对 `add` 的调用地址修正为真正的函数地址。

---

## 6. 符号是什么

符号可以简单理解为：

```text
函数、全局变量、静态变量等在目标文件里的名字和地址信息。
```

例如：

```cpp
int g = 10;

int add(int a, int b) {
    return a + b;
}
```

编译后可能产生符号：

```text
g
add(int, int)
```

可以使用 `nm` 查看：

```bash
nm main.o
```

常见符号类型：

```text
T add      // Text 段中的函数定义
D g        // Data 段中的已初始化全局变量
B x        // BSS 段中的未初始化全局变量
U printf   // Undefined，当前文件引用了但未定义
```

其中 `U` 表示当前目标文件引用了这个符号，但是没有定义，需要链接器去其他目标文件或库中查找。

## 7. 静态库

### 7.1 静态库是什么

静态库本质是：

```text
一堆 .o 目标文件打包在一起的归档文件。
```

Linux 下通常是：

```text
libxxx.a
```

Windows 下通常是：

```text
xxx.lib
```

例如：

```bash
g++ -c add.cpp -o add.o
g++ -c sub.cpp -o sub.o

ar rcs libmath.a add.o sub.o
```

这样就生成了静态库：

```text
libmath.a
```

---

### 7.2 使用静态库

```bash
g++ main.cpp -L. -lmath -o main
```

含义：

```bash
-L.
```

表示在当前目录查找库。

```bash
-lmath
```

表示查找：

```text
libmath.a
```

或者：

```text
libmath.so
```

---

### 7.3 静态库链接原理

假设：

```cpp
// main.cpp
int add(int, int);

int main() {
    return add(1, 2);
}
```

静态库 `libmath.a` 里面包含 `add.o`。

链接时：

```bash
g++ main.o -L. -lmath -o main
```

链接器发现 `main.o` 需要 `add`，于是从 `libmath.a` 中抽取包含 `add` 的目标文件，复制进最终可执行文件。

最终生成的 `main` 中已经包含了 `add` 的机器码。

所以运行时不再依赖 `libmath.a`。

---

### 7.4 静态库特点

优点：

```text
1. 部署简单，可执行文件包含所需代码。
2. 运行时不依赖外部库文件。
3. 启动时少一些动态装载开销。
```

缺点：

```text
1. 可执行文件体积较大。
2. 多个程序使用同一个静态库时，每个程序都各自复制一份代码。
3. 库升级后，需要重新链接生成可执行文件。
```

---

## 8. 动态库

### 8.1 动态库是什么

动态库是运行时可以被加载的共享库。

Linux 下通常是：

```text
libxxx.so
```

Windows 下通常是：

```text
xxx.dll
```

macOS 下通常是：

```text
libxxx.dylib
```

---

### 8.2 生成动态库

```bash
g++ -fPIC -shared add.cpp -o libmath.so
```

参数说明：

```text
-fPIC   生成位置无关代码
-shared 生成动态库
```

---

### 8.3 使用动态库

```bash
g++ main.cpp -L. -lmath -o main
```

此时链接器不会把 `libmath.so` 的全部代码复制进 `main`。

它只会在可执行文件中记录：

```text
运行时需要 libmath.so
需要其中的 add 符号
```

运行时由动态链接器加载动态库。

---

## 9. 动态库的运行时加载

程序启动时，操作系统的动态链接器会负责加载动态库。

Linux 上常见动态链接器：

```text
/lib64/ld-linux-x86-64.so.2
```

动态链接器主要负责：

```text
1. 找到程序依赖的动态库。
2. 把动态库映射到进程虚拟地址空间。
3. 解析动态库中的符号。
4. 修正函数调用地址。
5. 执行程序入口。
```

可以使用下面命令查看程序依赖了哪些动态库：

```bash
ldd main
```

示例输出：

```text
linux-vdso.so.1
libmath.so => ./libmath.so
libstdc++.so.6
libc.so.6
```

---

## 10. 动态库查找路径

如果运行时报错：

```text
error while loading shared libraries: libmath.so: cannot open shared object file
```

说明运行时找不到动态库。

常见解决方法如下。

### 10.1 设置 LD_LIBRARY_PATH

```bash
export LD_LIBRARY_PATH=.:$LD_LIBRARY_PATH
./main
```

表示运行时从当前目录查找动态库。

### 10.2 使用 rpath

编译时写入运行时搜索路径：

```bash
g++ main.cpp -L. -lmath -Wl,-rpath=. -o main
```

### 10.3 放到系统库路径

例如：

```text
/usr/lib
/usr/local/lib
```

然后执行：

```bash
sudo ldconfig
```

## 11. 为什么动态库需要 -fPIC

`-fPIC` 表示：

```text
Position Independent Code，位置无关代码。
```

动态库可能被不同进程加载到不同的虚拟地址。

例如：

```text
进程 A 中 libmath.so 加载到 0x7f100000
进程 B 中 libmath.so 加载到 0x7f300000
```

如果动态库代码依赖固定地址，就不利于重定位和共享。

`-fPIC` 生成的位置无关代码，可以让动态库被加载到不同地址仍然正常运行。

面试回答：

```text
动态库可能被多个进程加载到不同的虚拟地址空间中，-fPIC 生成位置无关代码，使代码不依赖固定装载地址，减少重定位开销，也便于多个进程共享动态库代码段。
```

## 12. 静态库和动态库对比

| 对比项 | 静态库 `.a` | 动态库 `.so` |
|---|---|---|
| 链接方式 | 链接时复制目标代码进可执行文件 | 链接时只记录依赖，运行时加载 |
| 可执行文件体积 | 较大 | 较小 |
| 运行时依赖 | 不依赖库文件 | 依赖动态库存在 |
| 部署 | 简单 | 需要处理库路径和版本 |
| 升级 | 需要重新链接程序 | 通常替换 `.so` 即可 |
| 内存共享 | 多个程序各有一份 | 多个进程可共享代码段 |
| 启动开销 | 相对小 | 需要动态加载和符号解析 |
| 适合场景 | 单文件部署、环境不可控 | 多程序共享、插件、系统库 |

## 13. 静态链接和动态链接的区别

### 13.1 静态链接

```bash
g++ main.o libmath.a -o main
```

最终结果：

```text
main 里面包含 libmath.a 中被用到的代码。
```

运行时：

```text
不需要 libmath.a。
```

---

### 13.2 动态链接

```bash
g++ main.o -L. -lmath -o main
```

最终结果：

```text
main 里面只记录需要 libmath.so。
```

运行时：

```text
必须能找到 libmath.so。
```

可以简单理解为：

```text
静态库：链接时把代码搬进可执行文件。
动态库：运行时把库加载进进程地址空间。
```

---

## 14. 链接顺序问题

Linux 下静态库链接顺序很重要。

错误示例：

```bash
g++ -L. -lmath main.o -o main
```

可能报错：

```text
undefined reference to `add`
```

正确写法：

```bash
g++ main.o -L. -lmath -o main
```

原因是 GNU ld 通常从左到右扫描。

如果先看到 `main.o`，发现它需要 `add`，后面看到 `-lmath` 时，就会从库里查找 `add`。

如果先看到 `-lmath`，此时还不知道需要 `add`，库被扫描过去后，再看到 `main.o`，就可能找不到符号。

经验规则：

```text
依赖别人者放前面，被依赖的库放后面。
```

例如：

```bash
g++ main.o -lA -lB -o main
```

表示：

```text
main 依赖 A，A 依赖 B。
```

## 15. C++ 符号名改编

C++ 支持函数重载，所以编译器会对函数名进行改编。

例如：

```cpp
int add(int, int);
double add(double, double);
```

这两个函数都叫 `add`，但参数不同。

为了让链接器区分它们，C++ 编译器会把函数名编码成不同的符号名。

例如：

```text
_Z3addii
```

这叫：

```text
Name Mangling，符号名改编。
```

如果想导出 C 风格符号，可以使用：

```cpp
extern "C" int add(int a, int b);
```

这样导出的符号名就是：

```text
add
```

这在 C++ 动态库给 C、Python、Go 等语言调用时很常见。

## 16. 动态库运行时加载：dlopen

动态库除了可以在程序启动时自动加载，也可以在运行时手动加载。

典型场景：

```text
1. 插件系统
2. 策略模块热加载
3. 不同券商 API 动态接入
4. Python ctypes 调 C/C++ 动态库
```

示例：

```cpp
#include <dlfcn.h>
#include <iostream>

using AddFunc = int(*)(int, int);

int main() {
    void* handle = dlopen("./libmath.so", RTLD_LAZY);
    if (!handle) {
        std::cerr << dlerror() << std::endl;
        return 1;
    }

    auto add = reinterpret_cast<AddFunc>(dlsym(handle, "add"));
    if (!add) {
        std::cerr << dlerror() << std::endl;
        return 1;
    }

    std::cout << add(1, 2) << std::endl;

    dlclose(handle);
}
```

动态库中需要导出 C 风格符号：

```cpp
extern "C" int add(int a, int b) {
    return a + b;
}
```

编译：

```bash
g++ -fPIC -shared math.cpp -o libmath.so
g++ main.cpp -ldl -o main
```

## 18. 常见报错

### 18.1 undefined reference

错误示例：

```text
undefined reference to `add(int, int)'
```

含义：

```text
编译通过了，但链接阶段找不到 add 的定义。
```

常见原因：

```text
1. 只声明了函数，没有实现。
2. 实现文件没有参与链接。
3. 静态库链接顺序错误。
4. 函数签名不一致。
5. C/C++ 符号名不一致，缺少 extern "C"。
```

---

### 18.2 multiple definition

错误示例：

```text
multiple definition of `add(int, int)'
```

含义：

```text
同一个符号被多个目标文件定义了。
```

常见原因：

```cpp
// add.h
int add(int a, int b) {
    return a + b;
}
```

多个 `.cpp` include 后，每个 `.o` 都有一个 `add` 定义。

解决方式：

```cpp
// add.h
int add(int a, int b);
```

```cpp
// add.cpp
int add(int a, int b) {
    return a + b;
}
```

或者：

```cpp
inline int add(int a, int b) {
    return a + b;
}
```

---

### 18.3 cannot open shared object file

错误示例：

```text
error while loading shared libraries: libxxx.so: cannot open shared object file
```

含义：

```text
编译链接成功，但运行时找不到动态库。
```

解决：

```bash
export LD_LIBRARY_PATH=/path/to/lib:$LD_LIBRARY_PATH
```

或者：

```bash
g++ main.cpp -L/path/to/lib -lxxx -Wl,-rpath=/path/to/lib -o main
```

---

### 18.4 file format not recognized

错误示例：

```text
file format not recognized
```

可能原因：

```text
1. 误把源码当库链接。
2. 库文件损坏。
3. 平台架构不一致，例如 x86_64 程序链接 ARM 库。
4. 32 位和 64 位混用。
```

可以检查：

```bash
file libxxx.so
file main.o
```

## 19. 头文件和库文件的关系

一个非常重要的点：

```text
include 了头文件，不等于链接了库。
```

头文件负责告诉编译器：

```text
有哪些函数、类、变量可以用，它们的声明是什么。
```

库文件负责提供真正的实现：

```text
这些函数的机器码在哪里。
```

例如：

```cpp
#include "add.h"
```

只能让编译器知道：

```text
add(int, int) 是一个合法函数。
```

但是链接阶段仍然需要：

```bash
g++ main.o add.o -o main
```

或者：

```bash
g++ main.o -L. -lmath -o main
```

否则会报：

```text
undefined reference to `add(int, int)'
```

总结：

```text
头文件负责声明。
库文件负责实现。
链接器负责把声明和实现对上。
```

## 20. 面试回答模板

### 20.1 程序编译过程是什么？

可以回答：

```text
C/C++ 程序从源码到可执行文件一般经过预处理、编译、汇编、链接四个阶段。

预处理负责展开头文件、宏替换和条件编译；
编译阶段进行语法语义检查、优化并生成汇编；
汇编阶段把汇编代码转成目标文件；
链接阶段把多个目标文件和库合并，解析符号引用，完成地址重定位，最终生成可执行文件或动态库。
```

---

### 20.2 静态库和动态库有什么区别？

可以回答：

```text
静态库本质是一组目标文件的打包，链接时会把被使用到的目标代码复制进最终可执行文件，所以运行时不依赖静态库，但可执行文件体积较大，库升级需要重新链接。

动态库在链接时只记录依赖和符号信息，运行时由动态链接器加载到进程地址空间。多个进程可以共享动态库代码段，升级和插件化更方便，但部署时要处理库路径、版本兼容和运行时加载失败等问题。
```

---

### 20.3 为什么动态库要加 -fPIC？

可以回答：

```text
动态库可能被多个进程加载到不同的虚拟地址空间中，-fPIC 会生成位置无关代码，使库代码不依赖固定装载地址，减少重定位开销，也便于多个进程共享动态库代码段。
```

---

## 21. 核心总结

```text
1. .h 负责声明，.cpp 负责编译成 .o。
2. .o 里有机器码，但可能还有未解析符号。
3. 链接器负责把多个 .o 和库文件合并，解决符号引用。
4. 静态库 .a 是一堆 .o 的打包，链接时拷贝代码进可执行文件。
5. 动态库 .so 是运行时加载的共享对象，链接时只记录依赖。
6. include 头文件不等于链接库。
7. 编译错误多半是语法、类型、访问权限问题。
8. 链接错误多半是符号找不到、符号重复定义、库顺序错误。
```

最关键的理解：

```text
编译阶段只检查当前翻译单元是否能独立编译通过；
链接阶段才检查所有函数和全局变量的实现到底能不能找到。
```
